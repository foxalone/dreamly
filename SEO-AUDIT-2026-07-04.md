# Dreamly.art — Technical SEO Audit & Expansion Plan (2026-07-04)

## Part 1 — Current Implementation Audit

### 1. Indexable pages
- **278 dream pages**: 35 parent symbols + 243 variations, all statically generated from `lib/dream-dictionary.ts` via `app/dreams/[symbol]/page.tsx` (`dynamicParams = false`, full SSG).
- Plus `/` (homepage), `/dreams` (hub), `/invite`, `/app/shared`, `/app/map` → **~283 indexable URLs total**.
- The June 27 audit file (`dream-dictionary-cluster-audit.md`) is stale — it reports 15 parents / 98 pages; the dictionary has since grown to 35 / 278.

### 2. Internal linking structure
- Parent ↔ variation: bidirectional (parent lists all variations; variations link back to parent + up to 6 siblings).
- Related symbols: cross-cluster, validated for parent↔parent reciprocity at build time (good).
- Measured link floor: **min inbound = 4** (flying/falling variations), **min outbound = 9** (pregnancy variations). Well below a 15/15 target.
- No orphans (enforced by build-time validation) — but inbound distribution is heavily skewed toward parents and popular slugs.

### 3. Sitemap
- `app/sitemap.ts`: covers all 278 entries + 5 core URLs with real `lastmod` (per-entry `updatedAt`), sensible priorities. Solid.

### 4. Schema markup
- Symbol pages: `Article`, `FAQPage`, `BreadcrumbList`. Hub: `ItemList`. Homepage: `WebSite` + `SearchAction` + FAQ.
- Missing: `CollectionPage` for hubs, `ImageObject`/`image` on Article, speakable/none needed.

### 5. Metadata
- Unique `seoTitle`/`seoDescription` per entry, canonical URLs, OpenGraph + Twitter cards, per-symbol dynamic OG image (`opengraph-image.tsx`). `metadataBase` set. Good.

### 6–8. Symbols, variations, categories
- 35 parents / 243 variations across **8 categories** — but distribution is skewed: animals 14, places 11, movement 2, fear-nightmares 2, life-events 3, body 1, water 1, objects 1.
- **No category hub pages** — categories exist only as `#anchor` sections on `/dreams`. Category-level queries ("animal dreams meaning") have no dedicated landing URL.

### 9. FAQ generation
- 5 templated FAQs per entry, emitted as FAQPage JSON-LD + visible `<details>` markup. Works, but heavily templated (identical sentence frames across 278 pages).

### 10. Related-symbol logic
- Curated `relatedSymbols` per cluster, inherited verbatim by variations; reciprocity validated for parents. No algorithmic augmentation, so inbound counts are uneven.

### 11. Search
- Client-side fuzzy search over slugs/aliases on `/dreams`, deep-linkable via `?q=` (wired to `SearchAction` schema). Fine.

### 12. Technical issues limiting growth
1. **No category hubs** → category-intent queries unserved; crawl depth relies on one giant `/dreams` page.
2. **No A–Z index** → no guaranteed flat inbound link to every page.
3. **Link floor of 4 inbound / 9 outbound** → weak PageRank flow to long-tail variations.
4. **No combination pages** ("snake and water") → zero coverage of a large long-tail class.
5. **Template monotony** → all 278 pages share the same sentence frames; risk of "scaled content" quality dampening as the site grows. (Mitigated by unique focus phrases; flagged for future differentiation.)
6. **No images beyond OG** — emoji only; no Google Images surface, no `image` property in Article schema.
7. Missing high-volume parents (see gap analysis).

## Part 3 — Expansion Architecture (designed for 2k/5k/10k pages)

All content remains data-driven in `lib/dream-dictionary.ts` (single source of truth); pages/sitemap/links generate automatically. Growth = adding seed rows, never new routes.

- **A. New parent symbols (this pass, 18):** fire, tornado, earthquake, storm, blood, hair, ex-partner, cheating, wedding, kissing, mother, ghost, demon, angel, rat, alligator, elephant, naked-in-public, work, phone (final list in implementation).
- **B. Variations:** ~7–8 per new parent, one URL each, unique focus text.
- **C. Category hubs:** `/dreams/categories/{slug}` — a dedicated statically generated hub per category. (Not `/dreams/{category}` because `water` the category collides with `water` the symbol; `categories/` prefix removes an entire collision class forever.)
- **D. Combination pages:** curated, semantically justified pairs only (e.g. snake+water, baby+death, snake+baby), generated as first-class entries at `/dreams/{a}-and-{b}` with dedicated combination content. No cartesian spam.
- **E. Authority hubs:** `/dreams/a-z` (A–Z index of every page — guarantees ≥1 flat inbound link to all), `/dreams/most-common`, `/dreams/biblical`, `/dreams/islamic`, `/dreams/spiritual`, `/dreams/nightmares`.

### Religious clusters decision
Each symbol page already carries substantive Islamic / Biblical / Spiritual sections at a single canonical URL. Splitting them into `/biblical/snake-dream` etc. today would create three near-duplicates per symbol from the same template — a Panda-class risk, and it triples the thin-content surface. Correct sequencing: build `/dreams/biblical|islamic|spiritual` **hub pages** now (unique intro content + deep links to every symbol's faith section); split into per-faith URL trees only when per-faith content is written independently per symbol. Architecture reserves the URL space; nothing built now blocks it.

### Internal linking machine (automated, no hardcoding)
1. Parent ↔ variation (exists).
2. Variation ↔ variation: siblings (exists) + **deterministic ring links** — entries sorted, each page links to the next 6 in the ring → every page gains exactly 6 inbound + 6 outbound, evenly distributed.
3. Category ↔ symbol: category hub lists every entry in category; every symbol page links its category hub.
4. Category ↔ category: hub-to-hub nav on each category page.
5. Combination ↔ parent: combos link both parents + their variations; parents list their combos.
6. Hub layer: A–Z + footer nav in dictionary layout link hubs site-wide.
Result: guaranteed ≥15 inbound and ≥15 outbound per page (verified programmatically post-build).

### Image SEO
No unique raster images exist; per-symbol OG images are generated. This pass: expose the OG image in Article schema (`image`) and metadata alt text. Real per-symbol illustrations remain a future content investment (needed for genuine Google Images visibility).
